# SpecBridge MCP Agent Guide

This file gives coding agents project-specific context. Follow higher-priority system/developer/user instructions first, then use this guide for repository work.

## Project intent

SpecBridge MCP is a clone-and-own Model Context Protocol server for API contract intelligence. It reads OpenAPI-compatible documents, including OpenAPI documents emitted by Huma services, and exposes deterministic contract facts to agents through MCP tools.

The project should stay repository-first and small. Do not turn it into a broad SDK generator, hosted service, or downstream-code mutator unless the user explicitly asks for that direction.

## Current architecture

- `src/index.ts` owns process startup, CLI/env transport parsing, stdio transport, and stateful/stateless HTTP transport. Keep service-specific logic out of this file.
- `src/mcp/register-tools.ts` owns MCP tool registration and request/response shaping. Tools should remain thin wrappers over deterministic service modules.
- `src/openapi/config.ts` owns backend registry loading, precedence, default backend selection, and spec cache timing config.
- `src/openapi/spec.ts` owns contract document loading, JSON/YAML parsing, Swagger UI extraction, endpoint lookup, validation facts, referenced schemas, and TypeScript DTO declaration emission.
- `src/openapi/types.ts` owns shared OpenAPI/Huma-compatible contract types.
- `openapi.backends.example.json` is the committed demo registry. Keep it safe and public.
- `openapi.backends.json` is ignored local config. Never commit private service URLs or credentials there.
- `build/` is generated by `pnpm build`; do not edit it by hand.

## Huma support posture

Huma is supported through the OpenAPI-compatible JSON/YAML documents that Huma services emit. The Huma path has smoke coverage, but not broad conformance coverage across many real Huma services. Phrase docs and tool descriptions as supported but maturing, and prefer adding fixtures/tests when expanding Huma-specific behavior.

Some internal file names and env vars say `openapi` because that is the wire format consumed by the parser. Do not rename everything casually; aliases and docs are usually safer than churn.

## Local backend registry rules

- Keep `openapi.backends.json` ignored.
- Use `cp openapi.backends.example.json openapi.backends.json` for local setup.
- Use `OPENAPI_BACKENDS_FILE` or `OPENAPI_BACKENDS` for alternate registries in tests or host-specific setups.
- Use stable lowercase backend IDs; agents pass `backendId` into every tool call.
- Use `domainHints` for endpoint-not-found suggestions when a clone has multiple services.

## How to add extra service layers

Prefer configuration-only extension first. If another service exposes OpenAPI or Huma-generated OpenAPI, add it to `openapi.backends.json`, `OPENAPI_BACKENDS_FILE`, or `OPENAPI_BACKENDS` instead of adding code.

Add code only when the service needs custom auth, discovery, transformation, or new MCP tools. Use this shape:

1. Keep transport in `src/index.ts` unchanged.
2. Add service-local types in `src/<service>/types.ts`.
3. Add service-local configuration/env parsing in `src/<service>/config.ts`.
4. Add deterministic service logic in `src/<service>/<domain>.ts`.
5. Register MCP tools in a small `src/mcp/register-<service>-tools.ts` helper if `register-tools.ts` would become too large.
6. Import the helper from `src/mcp/register-tools.ts` and call it from `registerTools`.
7. Return both `content` and `structuredContent` from each MCP tool.
8. Add unit tests for parsing/edge cases and an MCP smoke test for new tools.

Do not add a dependency unless the user explicitly approves it or the task clearly requires it. Prefer the existing `yaml`, `zod`, TypeScript, and Node standard library stack.

## Testing and quality gates

Run these before claiming completion for code or docs that affect checked files:

```bash
pnpm check
pnpm typecheck
pnpm test
```

Use narrower commands while iterating, but final evidence should include the full set above unless the user asks for a smaller change and you report the gap.

## Style and output conventions

- Keep diffs small and reversible.
- Prefer deterministic contract facts over best-effort inference.
- Label best-effort output clearly so downstream agents do not treat it as source-of-truth schema data.
- Keep README user-facing and this AGENTS.md agent-facing.
- Do not initialize or use shadcn tooling unless the user explicitly asks for shadcn UI/component work.
